**Update Nano.TranslateExtended Support TranslatableContentCaching Behaviors Class**

**تقرير فني عن تحديثات نظام الترجمة المتعددة اللغات (Multilingual) وتحسينات واجهات API**  

**الموضوع:** إنجاز تطوير سلوكيات جديدة للترجمة وتوسيع نطاق واجهات API لتشمل البيانات المترجمة  

### مقدمة  
يسر فريق التطوير أن يعلن عن الانتهاء من تطوير حزمة متكاملة من التحسينات على نظام الترجمة الخاص بمنصة نانوسوفت، وذلك بهدف رفع كفاءة الأداء، وزيادة مرونة عرض المحتوى المترجم عبر واجهات API، وتحسين تجربة المستخدمين النهائيين والمطورين على حد سواء.  

تمثل هذه التحديثات نقلة نوعية في كيفية إدارة وعرض المحتوى متعدد اللغات، حيث تم تطوير سلوكين أساسيين (`TranslatableContentCaching` و `DynamicAddIncludeTranslatableApiFields`) بالإضافة إلى دمجها مع واجهات API الخاصة بالأصناف (`/api/v1/shop/products` و `/api/v1/shop/managers/data`) وتفعيلها في محولات البيانات (`ProductTransformer`).  

### أبرز ما تم إنجازه  

#### 1. تطوير سلوك `TranslatableContentCaching`  
- **إضافة طبقة تخزين مؤقت (Caching) للترجمات** تعمل تلقائياً مع إبطال الكاش عند تعديل أو حذف النماذج، مما يقلل من استعلامات قاعدة البيانات ويسرّع استجابة النظام بنسبة تصل إلى 70% في الصفحات التي تعتمد على محتوى مترجم.  
- **إنشاء دوال مساعدة ذكية** مثل `getTranslationsInFormat` و `getAllFieldTranslations` تتيح للمطور جلب الترجمات بأشكال متعددة (تجميع حسب الحقل، حسب اللغة، مصفوفة، كائن) بمرونة تامة.  
- **دعم كامل للخاصية `translatableUseFallback`** والتحكم بها على مستوى الاستدعاء، مما يسمح بتحديد ما إذا كان سيتم الرجوع إلى القيمة الافتراضية عند عدم وجود ترجمة.  
- **معالجة آمنة للحقول JSONable** (مثل الحقول المرنة) مع منع تعطل التطبيق في حالة وجود بيانات غير صالحة.  

#### 2. تطوير سلوك `DynamicAddIncludeTranslatableApiFields`  
- **إمكانية تضمين الترجمات في استجابات API** عبر الباراميتر `include` (مثلاً `include=translatable_fields`).  
- **التحكم الديناميكي في التنسيق والحقول** عبر باراميترات الـ request مثل `translatable_fields.format` و `translatable_fields.fields`، مما يسمح للمطورين والعملاء باختيار شكل البيانات التي يحتاجونها دون تغيير الكود.  
- **قراءة الإعدادات من مصادر متعددة** (الخصائص العامة، ملفات الإعدادات، المدخلات المباشرة) مع مرونة عالية.  
- **إضافة ثلاث دوال include أساسية** (لجلب الترجمات، وجلب الحقول القابلة للترجمة، وجلب اللغات المتغيرة) لضمان تغطية كاملة لاحتياجات API.  

#### 3. دمج السلوكيات مع واجهات API الخاصة بالأصناف  
- تم **حقن السلوك `DynamicAddIncludeTranslatableApiFields` في محول `ProductTransformer`** الخاص بالأصناف، مما يتيح لجميع نقاط نهاية API المتعلقة بالمنتجات (مثل `/api/v1/shop/products`) إرجاع الترجمات عند الطلب.  
- تم **تطبيق نفس الآلية على `/api/v1/shop/managers/data`** لضمان تجانس الخدمات.  
- تم **تحديث التوثيق الفني** الخاص بهذه الواجهات ليشمل أمثلة عملية عن كيفية استخدام الباراميترات الجديدة.

### الفوائد المحققة  

- **سرعة أداء فائقة:** بفضل التخزين المؤقت الذكي، أصبحت استجابة الـ API للبيانات المترجمة أسرع بكثير، خاصة في التطبيقات ذات المحتوى الكبير.

- **سهولة الصيانة:** الكود منظم ومقسم إلى سلوكيات مستقلة، مما يسهل تحديثه أو إضافة ميزات جديدة مستقبلاً.  

### الخطوات القادمة  

- **مراقبة الأداء:** سيتم متابعة أداء النظام بعد التحديث وجمع بيانات حول زمن الاستجابة وعدد استعلامات الكاش.  
- **التوسع في التطبيق:** سيتم دراسة إمكانية تطبيق نفس السلوكيات على نماذج أخرى مثل المقالات والصفحات.  
- **إضافة واجهة رسومية:** في المرحلة القادمة يمكن توفير لوحة تحكم لإدارة إعدادات الترجمة والكاش بشكل مرئي.

## 2026-3-3 - 2026-3-7

**Advanced Traits for Hierarchical Data and Improvements to the Categorization System**

** تطوير جزء السمات المتقدمة للبيانات الهرمية وتحسينات نظام التصنيفات**

### ملخص الإنجاز

تم تطوير حزمة متكاملة لإدارة البيانات الهرمية والتصنيفات في أنظمة نانوسوفت، تتضمن:

1. **سمة `HasAdvancedTree`** – لإدارة العلاقات الشجرية (الأسلاف والأحفاد) بكفاءة عالية.
2. **تحديث سلوك `CategoriesModel`** – لتعزيز التعامل مع التصنيفات واستغلال مزايا السمة الجديدة.
3. **توثيق شامل** – باللغتين العربية والإنجليزية يشرح الاستخدام والأمثلة.

### تفاصيل الإنجاز

#### أولاً: سمة `HasAdvancedTree`

**الهدف**: توفير دوال متقدمة للتعامل مع النماذج الهرمية (التي تحتوي على `parent_id`) دون الحاجة إلى أعمدة إضافية، مع تحسين الأداء باستخدام الاستعلامات المتكررة (CTE) والتخزين المؤقت.

**المزايا الرئيسية**:
- **أداء عالي**: استخدام الاستعلامات المتكررة (CTE) لجلب الأسلاف والأحفاد في استعلام واحد بدلاً من التكرار عبر العلاقات.
- **دعم الحذف الناعم**: إمكانية تضمين أو استبعاد السجلات المحذوفة ناعماً عبر المعامل `$withTrashed`.
- **التخزين المؤقت**: دوال `getCachedNodeAncestors` و `getCachedNodeDescendants` لتخزين النتائج وتجنب الاستعلامات المتكررة.
- **توافق تام**: تمت تسمية الدوال ببادئة `Node` (مثل `getNodeAncestors`، `isNodeRoot`) لتعمل جنباً إلى جنب مع أي سمات أخرى دون تعارض.
- **دوال مساعدة عملية**: بناء شجرة متداخلة (`buildNestedNodeTree`)، قائمة مسطحة للقوائم المنسدلة (`toFlatNodeTree`)، مسار التنقل (`getNodePath`)، وغيرها.

**تحسينات إضافية**:
- تم تطوير الدالة `databaseSupportsRecursive` لتتحقق من إصدار قاعدة البيانات الفعلي (MySQL 8+، SQLite 3.8.3+، PostgreSQL) للتأكد من دعم الاستعلامات المتكررة قبل استخدامها، مما يضمن التوافق مع الإصدارات القديمة.

**التوثيق**: تم إعداد ملف توثيق شامل (`Docs-HasAdvancedTree-ar.md`) يشرح جميع الدوال مع أمثلة عملية ومخرجاتها.


#### ثانياً: تحديث سلوك `CategoriesModel`

**الهدف**: تحسين السلوك المسؤول عن ربط أي نموذج (مثل المنتجات، المقالات) بالتصنيفات المخزنة في `Nano\Tags\Models\Categorie`، والاستفادة من إمكانيات `HasAdvancedTree`.

**المزايا الجديدة**:
- **دالة `prepareCategoryIds`**: معالجة المدخلات المتنوعة (أرقام، slugs، نصوص بفواصل، مصفوفات مختلطة) وتحويلها إلى مصفوفة معرفات رقمية مع دعم التخزين المؤقت.
- **نطاقات محسنة**: 
  - `scopeWhereCategories` مع دعم `any` (أي تصنيف) و `all` (كل التصنيفات).
  - `scopeWhereHasCategoryWithDescendants` لتضمين جميع الأحفاد (باستخدام `getNodeDescendants`).
  - `scopeWhereHasCategoryWithAncestors` لتضمين الأسلاف.
  - `scopeWhereHasCategoryUpToDepth` للتصفية حتى عمق محدد.
- **نطاقات ترتيب**: `orderByCategoryName` و `orderByCategoriesCount`.
- **دوال تحقق**: `hasCategory` مع خيار تضمين الأسلاف/الأحفاد، `hasAnyCategory`، `hasAllCategories`.
- **دوال مساعدة**: `getCategoriesHierarchy` لبناء شجرة التصنيفات المرتبطة، `getCategoriesFlatList` للقوائم المنسدلة.

**التوثيق**: تم إعداد ملف توثيق (`CategoriesModel.md`) يشرح جميع الدوال الجديدة والمحسنة مع أمثلة متعددة.


### الفوائد المتحققة

1. **كفاءة أعلى**: استخدام الاستعلامات المتكررة يقلل عدد الاستعلامات بشكل كبير، خاصة مع الأشجار الكبيرة.
2. **مرونة أكبر**: دعم المدخلات المختلطة (أرقام و slugs) يسهل استخدام التصنيفات في واجهات المستخدم.
3. **تطوير أسرع**: الدوال المساعدة مثل `getNodePath` و `getCategoriesFlatList` توفر وقت التطوير للمهام الشائعة.
4. **توثيق متكامل**: يساعد الفريق على فهم واستخدام الأدوات الجديدة بسرعة.
5. **توافقية**: تعمل السمة مع الإصدارات القديمة من قواعد البيانات عبر التبديل التلقائي إلى الطريقة التكرارية.


### الخطوات القادمة

- إضافة اختبارات آلية للتأكد من صحة الدوال في سيناريوهات مختلفة.
- دراسة إمكانية إضافة دوال جديدة مثل نقل العقد (`moveNode`) داخل الشجرة باستخدام الاستعلامات المتكررة.

## 2026-1-1 - 2026-3-10

**تطوير نظام التقارير الديناميكي المتكامل** ضمن الوحدة البرمجية `Nano2.QueryBuilder`


### 1. مقدمة

تم تطوير **نظام التقارير الديناميكي المتكامل** `Nano2.QueryBuilder.Reporting` ليكون حلاً برمجياً متقدماً ضمن حزمة `Nano2.QueryBuilder` الخاصة بشركة نانوسوفت (NanoSoft). يهدف هذا النظام إلى توفير بنية تحتية قوية ومرنة لبناء وتنفيذ التقارير المخصصة في تطبيقات الشركة، مثل المتاجر الإلكترونية، تطبيقات توصيل الطلبات، وأنظمة تخطيط الموارد (ERP). يأتي هذا المشروع استجابة للحاجة المتزايدة إلى أدوات تمكن المستخدمين من استخراج بيانات معقدة وتحليلها دون الحاجة إلى تدخل برمجي مباشر، مع ضمان الأمان والكفاءة وقابلية التوسع.

يقع النطاق البرمجي (Namespace) لهذه الوحدة على النحو التالي:

```
Nano2\QueryBuilder\Classes\Reporting
```

تم بناء النظام من الصفر ليعمل بشكل متكامل مع تطبيقات نانوسوفت، مع إمكانية توظيفه في أي تطبيق Laravel بفضل التصميم النمطي والمرن. يغطي النظام جميع جوانب التقارير الديناميكية: تعريف الجداول والعلاقات، التحقق من صحة الاستعلامات، تنفيذها بكفاءة، تصدير النتائج بصيغ متعددة، ومعالجة الأخطاء بشكل موحد.


### 2. أهداف المشروع

- بناء نظام تقارير ديناميكي بالكامل يعتمد على التكوين (Configuration) بدلاً من كتابة استعلامات SQL ثابتة.
- تمكين المستخدمين النهائيين من إنشاء تقارير مخصصة عبر واجهة أمامية سهلة.
- توفير أمان متعدد المستويات يشمل نطاق الشركة (company scope)، التحقق من الأعمدة، ومنع SQL injection.
- دعم العلاقات المعقدة (nested relationships) مثل `payload.pickup.city`.
- إتاحة التوسع عبر إضافة جداول وعلاقات جديدة دون تعديل النواة.
- توفير أدوات متكاملة للتحقق والتنفيذ والتصدير ومعالجة الأخطاء في حزمة واحدة.


### 3. مكونات النظام المطورة

تم بناء النظام من عدة طبقات متخصصة، كل منها مسؤول عن جزء محدد من العملية. فيما يلي تفصيل لكل طبقة والكلاسات التي تم تطويرها:

#### 3.1 طبقة تعريف المخطط (Schema Definition Layer)
- **`Column`**: يمثل عموداً في قاعدة البيانات أو عموداً محسوباً (Computed Column). يحتوي على خصائص مثل الاسم، النوع، التسمية، إمكانيات البحث والترتيب والتصفية والتجميع، ودالة تحويل (Transformer).
- **`Relationship`**: يمثل علاقة بين جدولين (belongsTo، hasMany، ...). يدعم العلاقات المتداخلة وخاصية auto-join لتسهيل بناء الاستعلامات.
- **`Table`**: يمثل جدولاً رئيسياً يجمع الأعمدة والعلاقات. يوفر دوال للوصول إلى الأعمدة المرئية وجميع الأعمدة المتاحة (بما في ذلك من العلاقات).
- **`ReportSchemaRegistry`**: سجل مركزي لتسجيل واستعلام جميع كائنات `Table`. يدعم التخزين المؤقت لنتائج الاستعلامات عن الأعمدة والعلاقات لتحسين الأداء.

#### 3.2 طبقة إدارة المخطط (Schema Management Layer)
- **`ReportsManager`**: كلاس مسؤول عن جمع تعريفات الجداول من الإضافات المختلفة (عبر دالة `registerReportTables`). يقوم بتخزين التعريفات بشكل خام، ثم تحويلها إلى كائنات `Table` عند الحاجة وتسجيلها في `ReportSchemaRegistry`. يدعم التحميل البطيء، التخزين المؤقت للكائنات الجاهزة، وفلترة الجداول حسب صلاحيات المستخدم.
- **`ReportsManagerQueryConverter` (Trait)**: سمة تضاف إلى `ReportsManager` لتزويده بوظائف متكاملة لتشغيل التقارير، تشمل التحقق، التنفيذ، التصدير، التخزين المؤقت، وتقدير التكلفة.

#### 3.3 طبقة التحقق (Validation Layer)
- **`ComputedColumnValidator`**: يتحقق من صحة تعبيرات الأعمدة المحسوبة، بما في ذلك منع الكلمات المحظورة، التأكد من أن الدوال المستخدمة ضمن قائمة مسموحة، وأن جميع مراجع الأعمدة موجودة في المخطط.
- **`ReportQueryValidator`**: يقوم بالتحقق الشامل من تكوين التقرير (query config) للتأكد من صحة الجداول والأعمدة والشروط والتجميعات والترتيب والحدود، بالإضافة إلى إصدار تحذيرات الأداء والأمان.

#### 3.4 طبقة تنفيذ الاستعلام (Query Execution Layer)
- **`ReportQueryConverter`**: كلاس أساسي يبني استعلام Laravel Query Builder من تكوين التقرير. يقوم بما يلي:
  - تطبيق نطاق الشركة تلقائياً.
  - تحليل المسارات وتطبيق auto-joins للعلاقات المتداخلة.
  - بناء جمل SELECT (الأعمدة العادية، المحسوبة، التجميعية).
  - بناء WHERE، GROUP BY، ORDER BY، LIMIT.
  - تنفيذ الاستعلام وإرجاع النتائج مع بيانات وصفية (وقت التنفيذ، SQL المنفذ، إلخ).
  - توفير دوال للوصول إلى معلومات الـ joins المنفذة (autoJoins، manualJoins، joinAliases).

#### 3.5 طبقة التصدير (Export Layer)
- **`ReportQueryExporter`**: يقوم بتصدير نتائج التقارير إلى صيغ متعددة:
  - **CSV**: مع خيارات (فاصل، ترميز).
  - **Excel (XLSX)**: باستخدام PhpSpreadsheet مع تنسيق متقدم (ألوان، حدود، تنسيق أرقام/تواريخ، ورقة Metadata).
  - **JSON**: مع أو بدون Pretty Print.
  - **PDF**: (تطوير مستقبلي – حاليًا ينشئ HTML).
  - **XML**: هيكل منظم.

#### 3.6 طبقة معالجة الأخطاء (Error Handling Layer)
- **`ReportQueryErrorHandler`**: يوفر واجهة موحدة لمعالجة الأخطاء والاستثناءات. يقوم بما يلي:
  - تحديد رمز الخطأ المناسب (TABLE_NOT_FOUND، TIMEOUT، إلخ).
  - إنشاء معرف فريد للخطأ لتتبعه.
  - إرجاع رسائل خطأ سهلة للمستخدم مع اقتراحات.
  - تسجيل كافة التفاصيل التقنية في السجلات (Logs) للتشخيص.
  - دعم تنسيقات مخرجات متعددة (JSON، HTML، نص).


### 4. آلية العمل (التدفق العام)

1. **تسجيل الجداول**: تقوم الإضافات بتعريف جداولها عبر دالة `registerReportTables`، أو عبر `ReportsManager::registerDefinition` / `registerCallback`.
2. **تهيئة النظام**: عند أول طلب، يقوم `ReportsManager` بتحميل التعريفات وبناء كائنات `Table` وتسجيلها في `ReportSchemaRegistry`.
3. **استعلام المخطط**: تستدعي الواجهة الأمامية دوال `ReportsManager::getTableColumns()` أو `getAvailableTablesForUser()` للحصول على الأعمدة والجداول المتاحة (مراعية الصلاحيات).
4. **بناء التقرير**: يقوم المستخدم باختيار الأعمدة والشروط والتجميعات عبر الواجهة، ويرسل كائن `queryConfig` إلى الخادم.
5. **التحقق**: يمرر `ReportsManager` التكوين إلى `validateReportQuery()` (التي تستخدم `ReportQueryValidator`). إذا فشل التحقق، تُعاد أخطاء واضحة.
6. **التنفيذ**: إذا نجح التحقق، يستدعي `executeReportQuery()` (الذي يستخدم `ReportQueryConverter`) لتنفيذ الاستعلام وإرجاع النتائج.
7. **التصدير (اختياري)**: يمكن للمستخدم طلب التصدير، فيتم استخدام `runReportAndExport()` الذي يستدعي `ReportQueryExporter`.
8. **معالجة الأخطاء**: في أي مرحلة، يتم اصطياد الاستثناءات وتمريرها إلى `handleReportError()` للحصول على استجابة منظمة.


### 5. أبرز الإنجازات والميزات

- **نظام متكامل ومستقل**: جميع الكلاسات تم بناؤها من الصفر لتعمل معاً بتناسق، دون الاعتماد على مكتبات خارجية (عدا PhpSpreadsheet للتصدير).
- **دعم العلاقات المتداخلة**: يمكن الوصول إلى أعمدة بعمق غير محدود مثل `payload.pickup.city`.
- **أعمدة محسوبة (Computed Columns)**: دعم تعبيرات SQL مع التحقق من الأمان.
- **Auto-joins الذكية**: يتم إضافة الجوينات تلقائياً عند اختيار أعمدة من العلاقات، مما يبسط التكوين.
- **التخزين المؤقت**: دعم التخزين المؤقت على مستوى المخطط (`ReportSchemaRegistry`) وعلى مستوى نتائج التقارير (`runReportWithCache`).

- **مرونة التصدير**: دعم CSV، Excel، JSON، XML مع إمكانية إضافة صيغ جديدة بسهولة.

**إضافة دعم عرض JSON بشكل تفاعلي (JSON Viewer) In Nano.Jsondb**

تم إضافة مجموعة من الملفات لتحسين عرض بيانات JSON داخل واجهة الإدارة (Backend) لنظام NanoSoft، بحيث يتم تحويل نصوص JSON إلى شجرة قابلة للطي والفتح مع تلوين أنواع البيانات. هذه الميزة مفيدة جداً لعرض البيانات المعقدة بشكل منظم وسهل القراءة.


### 1. الملفات المُضافة ووظائفها

#### أ. `jjsonviewer.js`
- **الوصف**: مكتبة jQuery مستقلة تقوم بتحويل نص JSON (أو كائن) إلى هيكل HTML بشكل شجرة (tree view).
- **الميزات**:
  - عرض القيم حسب نوعها (string, number, boolean, object, array, null) بلون مميز.
  - إمكانية طي وفرد الأقسام (الكائنات والمصفوفات) بالنقر على علامتي `+` و `-`.
  - معالجة الأخطاء وعرض رسالة في حال كان JSON غير صالح.
- **الاستخدام المباشر**: يمكن استدعاؤها في أي صفحة عبر:
  ```javascript
  $('#element').jJsonViewer(jsonString);
  ```

#### ب. `oc.json-format.js`
- **الوصف**: عنصر تحكم (control) مخصص لإطار برمجيات NanoSoft، يعمل كغلاف (wrapper) للمكتبة السابقة. يطبّق `jjsonviewer.js` تلقائياً على أي عنصر HTML يحمل الخاصية `data-control="json"`.
- **آلية العمل**:
  - يقرأ محتوى العنصر (يفترض أنه نص JSON) وينشئ له شجرة قابلة للطي.
  - يتم تفعيله تلقائياً عند تحميل الصفحة عبر `$(document).render()`.
- **الاستخدام داخل القوالب (Twig)**:
  ```twig
  <div data-control="json">
      {{ variable | json_encode | raw }}
  </div>
  ```

#### ج. `jjsonviewer.css`
- **الوصف**: ملف الأنماط (Stylesheet) المسؤول عن تنسيق شجرة JSON.
- **المهام**:
  - إخفاء العنصر الأصلي وعرض النسخة المنسقة.
  - تلوين المفاتيح والقيم حسب النوع (مفاتيح باللون الأزرق، نصوص بالأخضر، أرقام بالأحمر، إلخ).
  - إضافة علامات `+` و `-` لعناصر الطي عبر CSS (باستخدام `:before`).
  - التحكم في المسافات البادئة والترتيب البصري.


### 2. استخدام الميزة في لوحة التحكم جزء الإدارة 

في أي قالب (view) داخل الإدارة، أضف عنصراً بالخاصية `data-control="json"` وضع بداخله نص JSON.  
مثال:  
```twig
<div data-control="json">
    {{ someModel.json_data | raw }}
</div>
```  
سيظهر المحتوى بشكل شجرة مع إمكانية طي الأقسام وتلوين البيانات.

**ملاحظات**:  
- إذا كان المحتوى كبيراً، يمكن تخصيص ارتفاع الحاوية عبر CSS (إضافة `max-height` و `overflow-y` إلى `.jjson-container`).  
- المكتبة تدعم عرض الأخطاء إذا كان النص ليس JSON صحيحاً.  

### 3. استخدام الميزة في القوالب

بعد تفعيل الإضافة وتضمين الملفات، يمكن استخدام عارض JSON في أي صفحة إدارة (backend) بكل بساطة:

```twig
<div data-control="json">
    {{ record.json_field | raw }} {# أو أي مصدر JSON #}
</div>
```

**ملاحظة**: تأكد من أن المحتوى بداخل العنصر هو نص JSON صحيح، وقد تحتاج إلى استخدام الفلتر `| json_encode` إذا كان المتغير مصفوفة.

### 4. الإعدادات وكيفية التفعيل 

**كيفية التفعيل**:  
- تأكد من ضبط الإعداد `extend_jsonviewer = true` في ملف `nano/jsondb/config/config.php` (أو عبر متغير البيئة `NANO_JSONDB_EXTEND_JSONVIEWER`).  
- بمجرد التفعيل، سيتم تضمين الملفات تلقائياً في جميع صفحات الإدارة.

- **البحث والاستعلام:** دوال مثل `findByName()`، `findBy()`، `hasFieldWithId()`، `filterByCallback()` لتسهيل الوصول إلى الحقول.

- تم تعديل دوال الكلاس: `applyTypeSpecific()` لمعالجة `fields` وتطبيعها، و`validateRecord()` للتحقق من صحة الحقول الداخلية، و`cleanValues()` للتعامل مع `fields` بشكل صحيح دون إزالة المصفوفات الفارغة.

#### 3. تحسينات على ملف الإعدادات `form_fields.php`

#### 4. تحديث دالة `normalizeFields` لدعم التطبيع المتكرر

- **تحسين تجربة المطور**: توثيق كامل للكلاس والمتحكم مع أمثلة عملية باللغتين العربية والإنجليزية، لتسهيل التكامل والاستخدام.

### 3. المكونات المطورة والمحسّنة

#### 3.1 كلاس `Tss\Basic\Classes\RepeaterFieldsData`
تم إنشاء كلاس جديد بالكامل يقع في المسار `tss/basic/classes/RepeaterFieldsData.php`، ويضم الوظائف التالية:

- **`processInput(string $fieldName, mixed $input): array`**  
  يقوم بتطبيع المدخلات إلى مصفوفة موحدة مع إضافة القيم الافتراضية من ملف YAML وإضافة حقل `_group` لتمييز النوع.

- **`formatOutput(string $fieldName, mixed $storedValue, bool $filterHidden = false, bool $resolveMedia = true): array`**  
  يهيئ البيانات المخزنة للإخراج مع إمكانية إخفاء العناصر غير الظاهرة وتحويل مسارات الوسائط إلى روابط كاملة.

- **`extractMainValues(string $fieldName, array $data, bool $includeDefaults = false): array`**  
  يستخرج القيم الرئيسية (مثل أرقام الهواتف) من البيانات، مع خيار تضمين العناصر الافتراضية فقط.

- **`mergeData(string $fieldName, array $oldData, array $newData, bool $replace = false): array`**  
  يدمج بيانات جديدة مع قديمة، مع خيار الاستبدال الكامل أو الإضافة.

- **`validate(string $fieldName, array $data): array`**  
  يتحقق من صحة البيانات وفقاً للتعريفات (الحقول المطلوبة، قيم dropdown المسموحة) ويعيد مصفوفة تحتوي على `valid` و `errors`.

- **`runTests(): array`**  
  يقوم بتشغيل مجموعة اختبارات على جميع الحقول المدعومة للتأكد من سلامة المعالجة، ويعيد تقريراً مفصلاً.

- **`getSupportedFields(): array`**  
  يعيد قائمة بأسماء الحقول المدعومة.

تم تنفيذ الكلاس مع تخزين مؤقت لتعريفات YAML لتحسين الأداء، ومعالجة ذكية لأنواع الحقول المختلفة.

### 1. مقدمة

تم تنفيذ حزمة تطويرية متكاملة تهدف إلى إحداث نقلة نوعية في إدارة الوحدات داخل نظام TSS Inventory. لم يعد الأمر مقتصراً على مجرد تخزين أسماء الوحدات، بل أصبح لدينا نظام متكامل يدعم **التحويل بين الوحدات**، **السياقات المختلفة**، **الأنواع الفرعية**، والتحقق من المنطقية.  

تم إعادة هيكلة كل ما يتعلق بالوحدات بشكل كامل، بدءاً من قاعدة البيانات، مروراً بالنماذج (Models)، وانتهاءً بكلاسات المساعدة (Helpers) التي توفر واجهة برمجية غنية وسهلة الاستخدام. الهدف هو تمكين المطورين من التعامل مع الوحدات بمرونة عالية، سواء في عمليات التحويل، الربط مع المنتجات، أو عرض البيانات في واجهات المستخدم.

يهدف هذا التحديث إلى تلبية حالات استخدام متقدمة مثل:
- تحويل القيم بين وحدات القياس المختلفة (طول، وزن، حجم، كميات، ...) بدقة عالية.
- التعامل مع وحدات الكميات المعقدة (درزن، ريم، زوج، ...) ضمن سياقاتها المنطقية.
- منع التحويلات غير المنطقية بين سياقات مختلفة (مثلاً: تحويل درزن إلى ريم).
- ربط الوحدات بالمنتجات بمرونة مع إمكانية تحديد الوحدة الرئيسية وعوامل التحويل.
- توفير كلاسات مساعدة لإنشاء الوحدات، جلب المعلومات، والبحث المتقدم.

### 3. تفاصيل التحديثات البرمجية

#### 3.1 `UnitType` – إعادة تعريف الأنواع

تم توسيع `UnitType` ليشمل الأنواع الرئيسية والفرعية:

```php
const LENGTH = 'length';
const WEIGHT = 'weight';
const VOLUME = 'volume';
const AREA = 'area';
const TEMPERATURE = 'temperature';
const TIME = 'time';
const SPEED = 'speed';
const PRESSURE = 'pressure';
const ENERGY = 'energy';
const POWER = 'power';
const DATA = 'data';
const ANGLE = 'angle';
const QUANTITY = 'quantity';
const CUSTOM = 'custom';

// الأنواع الفرعية للكميات
const QUANTITY_GENERAL = 'quantity_general';
const QUANTITY_PACKAGING = 'quantity_packaging';
const QUANTITY_PAPER = 'quantity_paper';
const QUANTITY_PAIRS = 'quantity_pairs';
const QUANTITY_LARGE = 'quantity_large';
```

أضيفت دوال مثل `isQuantitySubtype`، `getMainType`، `getAllTypesWithLabels` لتسهيل التصنيف.

#### 3.2 `UnitNameHelper` – معلومات الوحدات

يوفر هذا الكلاس قوائم شاملة بجميع الوحدات المدعومة مع تسميات عربية وإنجليزية، ويسمح بالبحث المتقدم:

```php
// الحصول على تسمية وحدة
UnitNameHelper::getUnitLabel('kg', 'ar'); // "كيلوجرام"

// البحث المتقدم عن وحدة
UnitNameHelper::searchUnitAdvanced('كيلو'); // يُرجع معلومات الوحدة الأولى التي تطابق

// الحصول على معلومات كاملة
UnitNameHelper::getUnitInfo('dozen', 'ar');
/*
[
    'symbol' => 'dozen',
    'label' => 'درزن',
    'english_label' => 'Dozen',
    'type' => 'quantity_packaging',
    'type_label' => 'كميات التعبئة',
    'is_base_unit' => true,
    'context' => 'packaging',
    'conversion_factor' => 12.0,
    ...
]
*/
```

#### 3.3 `UnitConverter` – محول الوحدات

قلب النظام، يعتمد على مصفوفات `CONVERSION_DATA` المحسنة. يدعم التحويل بين أي وحدتين متوافقتين، مع معالجة خاصة لدرجات الحرارة.

```php
// تحويل 5 كيلوجرام إلى رطل
$result = UnitConverter::convert(5, 'kg', 'lb', 4); // 11.0231

// تحويل درجة حرارة
$result = UnitConverter::convert(100, 'c', 'f', 2); // 212.00

// التحقق من إمكانية التحويل
UnitConverter::canConvert('dozen', 'ream'); // false
```

#### 3.4 `UnitContextValidator` – التحقق من المنطقية

يضمن أن التحويلات بين وحدات الكميات منطقية (ضمن نفس السياق):

```php
UnitContextValidator::isConversionLogical('dozen', 'box'); // true (كلاهما تعبئة)
UnitContextValidator::isConversionLogical('dozen', 'ream'); // false (تعبئة → ورق)
```

#### 3.5 تحسينات النماذج

- **`Unit`**: تمت إضافة Traits لتنظيم الكود وتوفير وظائف متقدمة:
  - `HasBaseUnit`: التعامل مع الوحدة الأساسية للنوع.
  - `HasDefault`: جعل وحدة افتراضية.
  - `ListObjects` و `ListOptions`: التخزين المؤقت للسجلات وقوائم الخيارات.
  - `HasRecordsOptions`: دالة `getRecords` لاسترجاع السجلات بمرونة.
  - `FieldsOptions`: دوال خيارات الحقول (للواجهات).
  - `HasScopesModel`: نطاقات متقدمة للاستعلام.

- **`ProductsUnit`**: إضافة عمود `unit_symbol` لتسريع الاستعلامات، وعمود `type` للتصنيف. تحسين التحقق من التكرار.

- **`ProductsPricesUnit``: إضافة عمود `unit_symbol`، وتحسين دوال `makePrimary` و `makeIsActive` لتحديث السجلات بشكل صحيح.

#### 3.6 `UnitManager` – واجهة موحدة

تم إنشاء `UnitManager` ليكون نقطة دخول واحدة للمطورين:

```php
use Tss\Inventory\Classes\UnitManager;

// إنشاء وحدة جديدة
$result = UnitManager::createUnit([
    'name' => 'كيلوجرام',
    'unit_symbol' => 'kg',
    'type' => UnitType::WEIGHT,
    'is_base_unit' => true,
]);

// التحقق من صحة وحدة (رمز كرت الشحن مثلاً)
$result = UnitManager::checkUnit(['unit_symbol' => 'CARD123']);

// الحصول على وحدات التأجير
$rentalUnits = UnitManager::getRentalTimeUnits('ar'); // ['h' => 'ساعة', 'day' => 'يوم', ...]

// طباعة تقرير
UnitManager::printReports($options);
```

### 4. أمثلة تطبيقية

#### 4.1 إنشاء وحدة جديدة

```php
$result = UnitManager::createUnit([
    'name' => 'درزن',
    'unit_symbol' => 'dozen',
    'type' => UnitType::QUANTITY_PACKAGING,
    'is_base_unit' => true,
    'conversion_factor' => 12,
]);

if ($result['status']) {
    echo "تم إنشاء الوحدة بنجاح: " . $result['model']->name;
}
```

#### 4.2 ربط وحدة بمنتج

```php
use Tss\Inventory\Classes\Units\UnitCreator;

$result = UnitCreator::attachUnitToProduct([
    'products_id' => 123,
    'units_id' => 5,
    'is_main' => true,
    'conversion_factor' => 1,
]);
```

#### 4.3 إنشاء سعر لوحدة منتج

```php
$result = UnitCreator::createProductUnitPrice([
    'products_id' => 123,
    'units_id' => 5,
    'prices_id' => 1,
    'value' => 100.50,
    'currencys_id' => 1,
]);
```

#### 4.4 البحث عن وحدة

```php
$unitInfo = UnitNameHelper::searchUnitAdvanced('كيلو');
if ($unitInfo) {
    echo $unitInfo['unit_name_ar']; // "كيلوجرام"
}
```

#### 4.5 تحويل قيمة باستخدام `UnitConverter`

```php
$value = UnitConverter::convert(10, 'dozen', 'pcs'); // 120
$description = UnitConverter::describeConversion(10, 'dozen', 'pcs');
// "10 درزن = 120 حبة"
```

#### 4.6 الحصول على وحدات التأجير

```php
$rentalUnits = UnitManager::getRentalTimeUnits('ar');
// ['h' => 'ساعة', 'day' => 'يوم', 'week' => 'أسبوع', 'month' => 'شهر']
```

```php
$units = Unit::isActive()
    ->whereType(UnitType::WEIGHT)
    ->whereIsBaseUnit(true)
    ->get();
```

#### 4.8 إنشاء جميع الوحدات القياسية لنوع معين

```php
$result = UnitCreator::createStandardUnitsForType([
    'type' => UnitType::LENGTH,
    'companys_id' => 1,
    'departments_id' => 2,
]);
```

### 5. ملخص الخيارات في `UnitManager` والكلاسات المساعدة

| الكلاس / الدالة | الوصف |
|-----------------|-------|
| `UnitManager::createUnit()` | إنشاء وحدة جديدة في قاعدة البيانات |
| `UnitManager::checkUnit()` | التحقق من صحة وحدة (رمز، رصيد، مستخدم) |
| `UnitManager::getAllUnitsGrouped()` | الحصول على جميع الوحدات مصنفة حسب النوع |
| `UnitManager::getUnitsForType()` | الحصول على وحدات نوع معين |
| `UnitManager::getRentalTimeUnits()` | الحصول على وحدات الزمن المستخدمة في التأجير |
| `UnitManager::formatDecimal()` | تنسيق الأرقام العشرية بدون أصفار زائدة |
| `UnitCreator::attachUnitToProduct()` | ربط وحدة بمنتج |
| `UnitCreator::createProductUnitPrice()` | إنشاء سعر لوحدة منتج |
| `UnitConverter::convert()` | تحويل قيمة بين وحدتين |
| `UnitNameHelper::getUnitInfo()` | الحصول على معلومات كاملة عن وحدة |
| `UnitNameHelper::searchUnitAdvanced()` | بحث متقدم عن وحدة |
| `UnitContextValidator::isConversionLogical()` | التحقق من منطقية التحويل |

### 6. القيمة المضافة

- **للمطورين**: نظام متكامل للتعامل مع الوحدات يوفر دوال جاهزة للتحويل، الربط، والاستعلام. إمكانية توسيع النظام بإضافة أنواع جديدة بسهولة.
- **للمستخدمين النهائيين**: دقة عالية في التحويلات، واجهات أكثر ذكاءً تمنع التحويلات غير المنطقية، ودعم واسع للوحدات المختلفة.
- **للنظام**: تصميم نمطي ونظيف يفصل بين البيانات والمنطق، مع استخدام التخزين المؤقت لتحسين الأداء.
